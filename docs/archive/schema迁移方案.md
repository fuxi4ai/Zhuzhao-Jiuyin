# 🔄 烛照九阴 · 新旧 Schema 迁移方案

> 状态: ⏳ 待哥哥拍板

---

## 现状问题

两套 schema 共存，数据分散：

| 体系 | 表数 | 数据量 | 问题 |
|------|------|--------|------|
| **v2 旧表** | 19 张 | ~3000 条 | 结构复杂、填充率低、字段冗余 |
| **v3 新表** | 5 张 + predictor_accuracy | ~1250 条 | 极简设计但数据未完整迁移 |

同一概念在多处存储（如"情绪周期"出现在 dim3 / cycle_quant / emotion_cycle），无法判断权威来源。

---

## 方案：渐进式迁移（推荐）

### 原则
1. **旧表只读不写** — 保留历史数据，不再写入新记录
2. **新表为唯一权威** — 所有新增数据写入 v3 表
3. **逐步迁移关键数据** — 先迁高频使用的，低频的保留在旧表
4. **不删旧表** — 至少保留一个季度

### Phase 1: 立即执行（本周）

| 动作 | 说明 | 工作量 |
|------|------|--------|
| 标记旧表为只读 | 在代码中注释所有旧表写入逻辑 | 小 |
| 迁移情绪周期数据 | dim3.emotion_stage + cycle_quant → emotion_cycle | 中 |
| 统一写入入口 | 所有新增产业信号只写 industry_signals | 小 |
| 迁移热点板块数据 | dim2.hot_sectors → hot_sectors | 小 |

### Phase 2: 数据对齐（1-2 周）

| 动作 | 说明 |
|------|------|
| 迁移产业信号 | dim2.sector_logic / dim2p → industry_signals 去重合并 |
| 迁移信息差 | dim2.supply_demand → information_gap |
| 迁移交易策略 | dim4_trade_plan → 保留（新表暂未设计对应表） |
| 建立视图兼容 | 对旧表的查询创建 VIEW 指向新表 |

### Phase 3: 清理（1 个月后）

| 动作 | 说明 |
|------|------|
| 验证数据一致性 | 新旧表数据对比 |
| 归档旧表 | 导出为 SQL 备份后 DROP |
| 更新所有代码引用 | 确保无代码再引用旧表 |

## 数据迁移优先级

### 高频查询（优先迁移）
1. ✅ **emotion_cycle** — 情绪周期（核心维度）
2. ✅ **industry_signals** — 产业信号（已 1163 条）
3. ✅ **predictor_accuracy** — 预测者准确率（刚建）
4. ⏳ **hot_sectors** — 热点板块（0 条，需从 dim2 迁移）

### 中频查询（Phase 2）
5. ⏳ **information_gap** — 信息差追踪（0 条，需推理构建）
6. ⏳ **daily_summary** — 每日摘要（1 条，需批量生成）

### 保留在旧表（暂不迁移）
- dim1_external_pricing — 外围数据，v3 暂无对应表
- dim4_trade_plan — 交易计划，v3 暂无对应表
- dim4_stock_analysis — 个股分析，v3 暂无对应表
- stock_tracking — 个股追踪（0 条）

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 迁移过程中数据丢失 | 高 | 迁移前备份、逐条验证 |
| 代码引用遗漏 | 中 | 全量 grep 搜索旧表名 |
| 新表字段不兼容旧数据 | 中 | 迁移脚本做字段映射 |
| 迁移耗时过长 | 低 | 分批执行，不阻塞业务 |
