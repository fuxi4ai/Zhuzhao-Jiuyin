## G001 — 板块概念混淆：光模块 ≠ CPO

**日期**: 2026-05-07  
**发现人**: 天哥

**问题**: 初始设计把"光模块/CPO"归为一个标准板块

**根因**: 缺乏光通信行业知识，误认为两者是同一概念的不同叫法

**解法**: 
- 光模块 = 可插拔方案（当前主流）
- CPO = 共封装光学（下一代技术路线）
- 严格区分为两个独立板块

**预防**: 涉及行业板块分类时，先查阅行业资料确认，不凭直觉合并

## G002 — SQL 字段名不存在

**日期**: 2026-05-07

**问题**: Phase 3 批量导入时报错 `table dim3_sentiment_tech has no column named key_points`

**根因**: 脚本中使用了不存在的字段名 `key_points`，实际应使用 `support_level`

**解法**: 修正 INSERT 语句，使用正确的字段名

**预防**: 写 SQL 前先 `PRAGMA table_info(tablename)` 检查字段

## G003 — SQL 歧义列名

**日期**: 2026-05-07

**问题**: JOIN 查询时报错 `ambiguous column name: date`

**根因**: 多表 JOIN 时未指定表名前缀

**解法**: 使用 `rd.date` 或 `d3.date` 明确指定来源表

**预防**: JOIN 查询中所有字段都加表名前缀

---

## G004 — UNION ORDER BY 不匹配

**日期**: 2026-05-07

**问题**: `1st ORDER BY term does not match any column in the result set`

**根根**: UNION 后 ORDER BY 使用了列名，但结果集结构不一致

**解法**: 改用 `ORDER BY 1` (按列位置排序)

**预防**: UNION 查询的 ORDER BY 使用列位置而非列名

---

## G005 — Gateway 重启丢失 defaultModel

**日期**: 2026-05-07

**问题**: 每次 Gateway 重启后 `defaultModel` 字段被清空，API 回退到 `gateway/jarvis`

**根因**: Gateway 重启时的配置写回机制会丢失部分字段

**解法**: 创建恢复脚本 `restore_coding_plan.sh`，重启后自动修复

**预防**: 
- 不要用编辑器手动改配置（Gateway 有文件监控会覆盖）
- 用脚本改配置，改完立即重启
- 只修改 `agents.defaults.model.primary`，不要动 `models.json` 的 `defaultModel`（会触发错误）

## G006 — openclaw.json 多了一层嵌套

**日期**: 2026-05-07

**问题**: Gateway 重启失败，报错 `agents: Unrecognized key: "model"`

**根因**: `setdefault` 创建了错误的嵌套路径，在 `agents` 下多了 `model` 键（正确路径是 `agents.defaults.model`）

**解法**: 删除多余的 `agents.model` 键

**预防**: 修改配置前先打印完整结构确认路径

## G007 — 重复数据导入

**日期**: 2026-05-07

**问题**: Phase 3 导入后 Dim3/Dim4 表出现 107 条重复记录（覆盖率超过 100%）

**根因**: Phase 2 和 Phase 3 导入时日期有重叠，使用了 `INSERT OR REPLACE` 但不同表的 REPLACE 条件不一致

**解法**: 执行去重 SQL，保留每个日期最新的 rowid

**预防**: 
- 批量导入前先检查日期是否已存在
- 定期运行去重检查
- 统一使用日期作为主键约束

## G008 — 量化系统指标不足导致偏差

**日期**: 2026-05-07

**问题**: 情绪周期双轨制对比，量化系统 91 条全部判定为"冰点"

**根因**: 7 个指标中只有 3 个有数据（涨停/跌停/量能），缺失 4 个关键指标导致总分偏低

**解法**: 标记为 Phase 5 待办，从 PDF 原文提取缺失指标

**预防**: 设计量化系统时先确认数据可用性，不要假设有数据

## G009 — NLP 提取噪音大

**日期**: 2026-05-07

**问题**: 四维度课件提取的"主线"字段包含大量无关文字（如"本次直播开篇明确了..."）

**根因**: 正则匹配范围太宽，没有在对应章节内提取

**解法**: 增加上下文窗口判断，只在四维度标题附近提取

**预防**: NLP 提取后做长度过滤，太长的说明没提取到关键点

## G010 — 文本标准化误替换

**日期**: 2026-05-07

**问题**: `standardize_text("关注光模块和CPO方向，锂电池和储能板块")` 输出乱码

**根因**: "储能" 匹配到 "新能源电池" 的别名链，导致重复替换产生错误文本

**解法**: 标注已知限制，推荐使用 `extract_sectors()` 提取而非 `standardize_text()` 替换

**预防**: 文本替换类函数要特别注意子串匹配问题

## G011 — API 切换教程版本混乱

**日期**: 2026-05-07

**问题**: 多次收到不同版本的 API 切换教程，内容相互矛盾

**根因**: 不同 AI（白泽、烛阴）各自总结，没有统一版本管理

**解法**: 以最终验证通过的版本为准，更新统一脚本

**预防**: 教程类文档需要版本号 + 最后验证日期，过期内容标注废弃

## G012 — tushare 频率限制

**日期**: 2026-05-07

**问题**: limit_list_d 接口 200 次/分钟限制，批量拉取 350 天数据时触发

**根因**: 该接口是高频消耗型，每次调用消耗 1 次频率

**解法**: 
- 增加 time.sleep(0.5) 间隔
- 剩余 150 条后续分批补拉
- 已获取 200 条（57% 覆盖）

**预防**: 批量拉取前先计算所需频率，考虑分批执行或使用后台任务

## G013 — 量化阈值需适配实际指标数

**日期**: 2026-05-07

**问题**: 原阈值基于 7 指标满分 70 分设计，当前只有 4 指标满分 40 分

**根因**: 阈值设计与实际可用指标数不匹配

**解法**: 按实际指标数调整阈值（冰点≤10/复苏 11-20/火热 21-30/退潮≥31）

**预防**: 阈值设计时考虑指标可用性，动态调整
